## 📌 1. Introduction

Dokumen Test Plan ini menetapkan strategi, ruang lingkup, sumber daya, dan jadwal pengujian untuk **Projek Warteg Babi**. Tujuan utamanya adalah memastikan sistem kasir (POS), pemesanan, kalkulasi harga, dan manajemen stok berjalan 100% akurat dan stabil saat digunakan dalam operasional warteg.

### 1.1 Scope (Cakupan Pengujian)

#### 🎯 In Scope (Fitur yang Diuji)

| Modul / Fitur | Akses Role | Skenario Pengujian Utama |
| --- | --- | --- |
| **Manajemen Menu & Harga** | Admin / Kasir | Tambah/Edit/Hapus menu (Babi balado,babi cabe ijo dll), ubah harga, toggle stok habis. |
| **Pemesanan / POS Kasir** | Kasir / Waiter | Input order dine-in/takeaway, variasi lauk, cetak struk, kalkulasi total + voucer |
| **Pembayaran (Payment)** | Kasir / Pelanggan | Transaksi Cash, QRIS, Transfer,voucer kalkulasi kembalian, serta void/pembatalan transaksi. |
| **Laporan Penjualan** | Owner / Admin | Rekap omzet harian/bulanan, laporan lauk terlaris, ekspor ke PDF/Excel. |
|  |  |  |

#### 🚫 Out of Scope (TIDAK Diuji)

- Jaringan internet / Wi-Fi lokasi warteg.
- Integrasi ke layanan pihak ketiga yang belum di-support versi ini.
- Perangkat keras kasir yang rusak fisik (printer rusak, tablet pecah).

---

### 1.2 Quality Objectives

> 🎯 **TARGET KUALITAS:**
> 
> - **Kalkulasi 100% Akurat:** Tidak boleh ada selisih 1 sen pun pada kalkulasi kasir/pembayaran.

## 🧪 2. Test Methodology

### 2.1 Overview & Levels

> ▶️ **Pendekatan Pengujian:** Agile Sprints
> 

> 🔽 **Functional Testing**
Memastikan fitur input pesanan, hitung kembalian, dan manajemen stok berjalan sesuai spec.
> 

> 🔽 **Integration Testing**
Memastikan transaksi di POS kasir langsung memotong stok di modul Dapur dan mencatat rekap di Laporan Owner.
> 

> 🔽 **User Acceptance Testing (UAT)**
Simulasi riil transaksi bersama Kasir/Owner warteg sebelum sistem dipakai untuk jualan.
> 

---

### 2.2 Bug Triage & Severity Definition

- **Critical / Blocker:** Aplikasi crash total, transaksi gagal tersimpan, kalkulasi harga salah. *(Fix ASAP)*
- **Major:** Fitur gagal tapi ada *workaround* (misal: gagal cetak resi otomatis tapi bisa cetak manual).
- **Minor:** Isu UI/UX, typo nama menu, warna button kurang pas.

---

### 2.3 Suspension & Resumption Criteria

> 🛑 **SUSPEND TEST:** Jika ditemukan bug *Blocker* (misal: Kasir tidak bisa pencet tombol bayar sama sekali) atau 40% test case gagal.
> 
> 
> ▶️ **RESUME TEST:** Setelah tim Dev merilis hotfix dan masalah *Blocker* terverifikasi aman.
> 

---

### 2.4 Test Completeness (Exit Criteria)

- [ ]  100% test case kritikal telah dieksekusi.
- [ ]  Pass rate pengujian mencapai minimal **95%**.
- [ ]  **0 Critical / Blocker Defects** yang masih *open*.
